# 电子凭证会计数据标准工具包分析 与 "内嵌 XBRL 优先"开发路径

日期：2026-07-23（同日二次修订：jar 深度拆解与配置驱动实证）
分析对象：`电子凭证会计数据标准推广应用工具包/1.基础工具包（推广应用版V1.0）`（财政部组织、免费开源、2025-03-23 版）

---

## 一、先纠正一个理解：不是每个 PDF 都能"提取出 XML"

实测本文件夹 36 张 PDF（PyMuPDF 检查内嵌附件）：

| 票种 | 内嵌 XBRL/XML？ | 实测 |
| --- | --- | --- |
| 铁路电子客票（乘车票、退票费票） | **有**（`rai_issuer_*.xml`，约 4KB） | 2/2 张都有 |
| 数电普票（蔬菜店、酒店、物流等 20 余张） | 无 | 0 附件 |
| 数电专票（建筑服务、机械租赁等） | 无 | 0 附件 |
| 航空行程单（东航、吉祥） | 无 | 0 附件 |
| 高德打车电子发票 PDF | 无 | 0 附件（高德的 XML 是独立文件随票交付） |

结论：XML 是开票瞬间税务平台生成的数据本体。除少数按标准封装内嵌实例的票种（铁路客票、财政票据、国库凭证、银行回单等），PDF 变不出 XML；数电票 XML 只能从受票方税务数字账户批量导出或开票方随票交付（如高德）获取——**PDF 文本层/OCR 路径仍是主流必修路径**。

## 二、工具包拆开看：机制、资产与实证（本轮新增）

### 2.1 xbrl-json-1.0.jar —— 配置驱动的 XBRL↔JSON 转换器

拆开 88 个条目，核心不是代码而是 **`xbrlJsonConfig/` 下 20 个配置 XML**（财政部官方字段字典）：

- 每个配置定义一种凭证：命名空间 URI（如 `…/taxonomy/2023-12-31/rai`）、元素名清单、字段类型（string/number/date/tuple 嵌套）、JSON 映射
- 例：`rai_issuer.xml` 定义铁路客票 40 个字段（TypeOfBusiness、Fare、TotalAmountExcludingTax、TaxRate、TaxAmount、ETicketNumber、NumberOfOriginalInvoice……）
- `VoucherType.properties` 注册 19 类票种——**比说明文档多出 `inv_tlf`（通行费发票，开具/接收）和 `inv_receiver`**，其中通行费是企业报销常见票种，恰好是我们 catalog 的缺口，工具包已备好官方字段定义
- Java 代码（pansoft 包）只是这些配置的解释器——**配置是数据，可以被任何语言复用**

### 2.2 pdf-xml-extractor-1.0.jar —— PDFBox 封装

1191 个条目里 95% 是 Apache PDFBox/fontbox 全家桶，真正的工具代码只有 `com.bosssoft.tool.util` 下 3 个类（PdfExtractor、IIOUtils、ZipUtil）。机制就是 PDFBox 的内嵌文件 API——与 PyMuPDF `embfile_*` 完全等价（已实测三行代码取出铁路票内嵌 XBRL）。**没有任何只有它能做而 PyMuPDF 做不到的事。**

### 2.3 命名空间版本问题（实测澄清）

配置写的是 `taxonomy/2023-12-31`，而用户 2026 年 6 月真票内嵌的是 `taxonomy/2021-11-30`——**元素名跨版本完全一致**。解析器应按元素名匹配，不与 taxonomy 版本硬绑定。

### 2.4 配置驱动解析实证（吃配置、不跑 Java）

用约 30 行 Python（xml.etree + PyMuPDF，均为现有依赖）做了完整实证：

1. 读 `rai_issuer.xml` 官方配置 → 载入 40 个字段定义
2. 从真实铁路票 PDF 取内嵌 XBRL → 按配置逐字段提取
3. 结果：命中 23/40 字段（其余为该票本就为空的字段），关键值全对（售/112.00/102.75/0.09/9.25）
4. 勾稽校验：102.75 + 9.25 = 112.00 == Fare，**PASS**

**结论：jar 的核心价值（官方字段字典 + 解析映射）可以零 Java 复刻；XBRL→官方 JSON 的中间转换层对我们无价值——我们直接映射到内部 InvoiceRecord 模型。**

## 三、铁路票内嵌 XBRL 的实测内容（关键证据）

**乘车票**（2026-06-30，票面只有"票价￥112.00"）：`TypeOfBusiness=售、TotalAmountExcludingTax=102.75、TaxRate=0.09、TaxAmount=9.25、Fare=112.00`
**退票费票**（2025-07，票面只有"退票费￥38.00"）：`TypeOfBusiness=退、TotalAmountExcludingTax=35.85、TaxRate=0.06、TaxAmount=2.15`

- 乘车票税额与 ÷1.09×9% 公式分毫不差（公式口径被官方数据反向证实）
- 退票费 6% 口径由发行方一手数据坐实，无需政策推断；`TypeOfBusiness`（售/退）是权威票型判定字段

## 四、利用方案：把工具包吃干榨净（不跑 Java）

按"资产复用"而非"代码复用"的原则：

| 工具包资产 | 利用方式 | 优先级 |
| --- | --- | --- |
| 20 个 xbrlJsonConfig/*.xml（官方字段字典） | 提取为项目 `config/xbrl/` 资产，写一个约 100 行的配置驱动解析器（xml.etree）；**新票种支持 = 放一个配置文件**（铁路、航空、数电普/专、通行费、财政票据等 20 类现成） | 高 |
| 内嵌实例文档提取机制 | PyMuPDF `embfile_get()` 已实现；OFD 内嵌（OFD 本身是 zip 容器）留接口不宣称（无样本） | 高 |
| 17 个官方 JSON 样例 | 作为解析器的 golden test 基准：解析输出与样例对拍，现成的回归测试集 | 中 |
| VoucherType 票种命名体系（含 inv_tlf 通行费） | 对齐 schema/router 的票种 ID，补上 catalog 缺口（通行费发票） | 中 |
| 数电票独立 XML（高德式 EInvoice 格式） | 其元素名（TotalAmWithoutTax/TotalTaxAm/…）与 einv 配置同源（总局 XML 与 MOF taxonomy 镜像设计）→ 与内嵌 XBRL **共用一套映射** | 高 |
| Java 运行时 / jar 本体 / XBRL→官方JSON 中间层 | 不用。本机无 Java；中间层对 InvoiceRecord 直映射无价值 | 不做 |
| 接收端会计入账字段（分录、三单匹配、抵扣属期） | 不取。超出采集台账定位，引入即过度设计 | 不做 |
| SM2 验签 | 不做（台账采集场景不需要密码学验真） | 不做 |

落地顺序建议：

1. `config/xbrl/` 放入 20 个官方配置（复制自工具包，注明来源与版本）
2. 写 `input_profile/embedded_xbrl.py`：PDF 内嵌检测（embfile）→ 按 configId 匹配配置 → 解析为 InvoiceRecord；铁路票全线切换到此通道（乘车/退票都读官方值）
3. 独立 XML 文件输入（P3b）复用同一解析器与映射
4. golden test：用官方 JSON 样例对拍回归

## 五、诚实的边界

- 内嵌 XBRL 目前只在铁路客票上实测确认（2/2）；财政票据、国库凭证、银行回单按标准应内嵌但本文件夹无样本——按"没有样本不宣称"原则，留接口不提前宣称
- 数电普票/专票/航空行程单 PDF 实测不内嵌（34/34），PDF 文本层/OCR 主路径修复（P1a）仍是必修课
- 工具包配置对应 taxonomy 2023-12-31 版，与真票 2021-11-30 版元素名一致（实测）；未来若元素名变更，按版本解耦的解析器只需更新配置文件
