## 二、中国流通发票现状调研结论（审查的事实基准）

### 2.1 数电发票已是绝对主力

- 依据国家税务总局公告 2024 年第 11 号，数电发票自 **2024-12-01 起全国正式推广**，与纸质发票同等法律效力
- 截至 2025 年 8 月底，使用数电发票的纳税人超 **6100 万户**，开票金额占全部发票金额 **超 90%**
- 2025-10-01 起铁路、民航**纸质报销凭证全面停用**；各地税控纸票加速退出（如贵州 2025 年停止有奖发票）

### 2.2 数电发票六大类别（官方口径）

电子发票（增值税专用发票）、电子发票（普通发票）、电子发票（航空运输电子客票行程单）、电子发票（铁路电子客票）、电子发票（机动车销售统一发票）、电子发票（二手车销售统一发票）；另有约 19 种特定业务标签票（建筑服务、成品油、通行费、不动产、农产品收购、差额征税、医疗服务等）。

关键票面特征（与审查直接相关）：

- **20 位发票号码，无发票代码、无校验码、无机器编号、单一联次**
- 交付版式为 **PDF / OFD 两种官方格式**；按《电子凭证会计数据标准》（财会〔2025〕9 号全国推广），**XML（含税务数字签名）是归档级格式**
- **电子发票（铁路电子客票）、电子发票（航空运输电子客票行程单）票面直接列明增值税税额**（财政部 税务总局公告 2026 年第 13 号：进项税额"为发票上列明或包含的增值税税额"）；只有公路、水路等其他客票仍按 3% 公式计算

### 2.3 仍在流通的其他票种

增值税纸票（专/普，存量过渡）、旧税控体系电子普票（PDF 版式）、卷式发票、通行费发票、通用机打/手工/定额发票（萎缩中）、机动车/二手车统一发票（已纳入数电）、财政票据（医疗收费票据、非税票据）、税收完税证明。

**结论：该 Skill 的票种布局（standard-invoice 双变体 + 交通/医疗/完税 9 个 schema）方向与真实流通结构基本吻合，覆盖了企业报销最常见票种。以下问题集中在数电票新特征的跟进细节上。**

## 三、问题清单（按严重度分级）

### S1 严重 —— 可能导致台账数据错误且用户无感知

**P1. 铁路/航空客票"凭空算税"，从不读取、也不核对票面已列明的税额**（2026-07-23 经三张真实发票实测后修订为 P1a / P1b，详见 `P1-修复方案.md`）

- P1a（航空行程单，确定性结构错误）：数电行程单的票价、燃油附加费是**不含税金额**，税额税率票面直接列明；当前提取器套用旧纸票逻辑（÷1.09×9% 拆分），实测两张真实机票：税额错 16.25 / 21.28 元，燃油附加费、民航发展基金整条丢失，台账总额比票面少 22%，号码取成 13 位电子客票号
- P1b（铁路客票）：乘车票票价为含税价、票面无税额行（2026-06-30 新开票核实，官方版式即无税额栏），÷1.09×9% 是官方口径（原"与政策矛盾"表述在此不成立，已修正）；真正的错误是**退票费误套 9%**（退票费非旅客运输服务，销方按 6% 计税——已被票面内嵌 XBRL 一手数据证实：`TaxRate=0.06、TaxAmount=2.15、TypeOfBusiness=退`），且公式计算值未做任何标注、静默"通过"。**2026-07-23 三次修订：铁路 PDF 内嵌官方 XBRL（实测 2/2），金额/税率/税额/售退类型全部可直读，修复方案升级为"内嵌 XBRL 优先"，详见 `电子凭证工具包分析.md`**
- 证据：`schemas/railway-ticket.yaml` 与 `schemas/air-ticket-itinerary.yaml` 中 `tax_total.aliases: []`；`scheme_extractors/railway.py` / `air_ticket.py` 不读票面税额；`validation/deductible_vat.py` 拆分后 `_refresh_invoice_totals()` 用自算值覆盖票面汇总，勾稽校验自证自洽
- 建议：航空数电版只读票面、票价+燃油并一行、民航基金单列不计税、简单相加对齐合计；铁路优先读票面内嵌 XBRL（无内嵌时公式兜底并标注"计算值"）；公式计算值一律标注来源。方案全文见 `P1-修复方案.md` 与 `电子凭证工具包分析.md`

**P2. 红冲（负数）发票被静态标记为正数发票**（2026-07-23 经真实红冲票二次实锤：易兴负数1/2 金额 -52966.99 / -96637.17，台账均写"是否正数发票=是"且状态"通过"）

- 证据：`config/template_profiles/current.yaml` 两个 sheet 的 `is_positive_invoice` 均为 `static: 是`；全 `src/` 检索不到任何 `is_positive_invoice` 赋值逻辑
- 后果：红字数电票、负数行全部写"是否正数发票=是"，台账数据错误
- 建议：按 `total_with_tax` 符号派生（负 → 否），一处小改

### S2 中等 —— 功能缺口或明显误导

**P3. 输入格式支持（2026-07-23 经真实 XML 样本实测后修订，拆为 P3a / P3b）**

**P3a. OFD：维持不支持，降级为已知边界**

- 用户决策：手头无足够 OFD 样本，且明确不要转换样本——"没有样本，再多宣称都是假的"。本项目不对未验证格式做任何宣称，符合同样的诚实标准
- 现状仅一处在意：`cli.py` 对 `.ofd` 报英文 "Unsupported input file type"。建议改为识别该后缀并输出中文指引（如"请改用 PDF 版式文件"），一行分支，不引入任何 OFD 解析依赖

**P3b. XML：建议从"加引导"升级为"正式支持"，优先级上调**

- 实测样本（高德打车数电发票 XML，13.49 元客运服务费）：Python 标准库 `xml.etree.ElementTree` 直接解析成功，**零新依赖**
- 字段与内部模型几乎一一对应：`EIid/InvoiceNumber`→数电发票号码，`Seller/Buyer Name/IdNum`→购销方，`IssueTime`→开票日期，`TotalAmWithoutTax/TotalTaxAm/TotalTax-includedAmount`→金额/税额/价税合计，`IssuItemInformation`→明细行（金额/税率/税额/价税合计逐行齐全，含折扣负数行）
- 可靠性本质高于 PDF 文本层提取：金额有语义标签无歧义、明细行天然分行、不经过别名/正则匹配——**P1 类"解析算错"问题对 XML 输入在原理上不存在**
- 附带解决其他问题的权威数据源：红蓝字标志（`InIssuType`，可为 P2 的"是否正数发票"提供权威值）、税收分类编码、简易计税标签、销方纳税人资格、主管税务机关、国密 SM2 签名（可验真防篡改）
- 定位克制：流通中 PDF 仍是主流、XML 获取渠道有限（税务数字账户可导出，部分平台如高德开票随 PDF 一并提供）——XML 支持是"加分项"，不改变 PDF 主路径，不做验签等增值功能
- 依据：《电子凭证会计数据标准》（财会〔2025〕9 号）明确数电发票的归档数据格式为 XML（含税务机关数字签名），PDF/OFD 只是其版式呈现

**P4. 校验通过即把置信度拉满到 ≥0.9**

- 证据：`record_validator.py` 校验无 issue 时 `confidence = max(confidence, 0.9)` 并写"通过当前字段校验"
- 后果：OCR 误读但恰好勾稽自洽的记录（如 8 误识为 0 且税额同错）被打上 0.9+ 置信度"通过"，用户不再复核。校验只能发现"不一致"，不能证明"识别对"——置信度上调方向反了
- 建议：校验只降不升；"校验通过"与"识别置信度"是两个维度，不应合并

**P5. 弱身份票种存在去重误杀**

- 证据：`ledger_rows._invoice_key = hash(seller_tax_id, invoice_no, invoice_date, total_with_tax)`。出租车、地铁定额、医疗票据普遍缺 `seller_tax_id`；若号码再漏识别，两张不同的票（同日同金额，如两张 5 元地铁票）`invoice_line_key` 相同 → 第二张被静默判为"疑似重复"跳过
- 建议：构成 key 的关键字段缺失过多时（如无号码又无税号），降级为"写入但提示疑似重复"，而不是跳过

**P6. 航空行程单日期取"页面最后一个日期"的启发式**

- 证据：`scheme_extractors/air_ticket.py` `dates[-1]`；schema 的 `invoice_date` 别名只有"填开日期 / DATE OF ISSUED"（旧版式），新数电行程单用"开票日期"
- 后果：新版行程单票面同时出现开票日期与航班日期时，可能抓到航班日期
- 建议：优先匹配"开票日期/填开日期"标签后的日期，标签缺失再退启发式

**P7. 明细 sheet 没有"开票日期"列**

- 证据：`current.yaml` detail sheet 字段无 `invoice_date`（summary sheet 有但 `required: false`）
- 后果：按明细行使用台账时无法按开票期间筛选、对账、归属月份——台账的基本使用场景受限
- 建议：模板与 profile 增加一列，零风险改动

### S3 轻度 —— 维护性与卫生问题

**P8. 死代码与废弃配置**

- `config/draft_ledger_mapping.yaml`：全仓库零引用，且其 `target_sheet: 发票台账` 与现行模板 sheet（发票信息采集）冲突，纯属误导
- `src/invoice_ledger/output/ledger_writer.py`（INDEX 自称"旧写入辅助逻辑"）、`feedback/`（占位未接线）、`schema/schema_bootstrapper.py`（无引用）
- 建议：删除，或文件头标 deprecated

**P9. "处理时间"列丢失时分秒**

- 证据：`ledger_rows._processed_date` 截断到日期（冒烟测试实测写入 `2026-07-23 00:00:00`）；同日多批次无法区分
- 建议：保留到分钟

**P10. 单省特判硬编码进通用 schema**

- 证据：`standard-invoice.yaml` `invoice_type_context.title_rules` 首条 `include_all: [山东增值, 普通发票]` → "山东增值税电子普通发票"。脏数据式存在，应移除或泛化

**P11. 版本号不一致**

- `pyproject.toml` `version = "0.0.0"`，与 `_meta.json` / `payload_publish.json` 的 1.0.1、`SKILL.md` 的 1.0.0 三者互不一致

**P12. 数电号码只验长度不验号段**

- `_is_standard_digital_invoice_number` 仅判 20 位纯数字；官方编码规则前 2 位为年度后两位、3-4 位为省级区划代码，可做廉价 sanity check（非必须）

### 已知边界（确认过，不算问题）

- 机动车/二手车销售统一发票等数电类别无 schema → 走"未建模+识别提示页"复核通道，符合 SKILL.md 声明的边界，合理
- 通行费发票无 schema → 同上
- 多页 PDF 按页拆单元、不支持一页多票/一票多页 → `runtime.yaml` 已声明 `complex_multipage_pdf: structured_unsupported`，合理
- `--update-existing` 明确拒绝未实现功能而非半实现，诚实设计，赞

### 审查中确认设计良好的点（防误报）

- 20 位数电号码正确识别并分列到"数电发票号码"列（实测）
- 去重三重判定（draft_row_id / invoice_line_key / 行指纹）对正常票种工作正常（实测）
- 金额勾稽容差 0.02 处理舍入合理；负数项目触发复核的机制存在
- OCR 缓存、证据仅对失败/复核单元保存、低 token 运行纪律，符合本地工具定位
- 安装器 GPU/CPU 自动选择、doctor 自检、`.venv` 隔离不污染系统 Python，均验证通过
