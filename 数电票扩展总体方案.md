# 数电票扩展总体方案（双轨制）

日期：2026-07-23
前提：保留旧版发票支持；以国家标准对齐数电票；不过度设计、无新依赖。

---

## 一、对"以官方JSON样例作为提取模板"的可行性判定

你的大方向**成立**，但有一处必须修正，否则按字面做会踩坑：

| 你的提议 | 判定 | 说明 |
| --- | --- | --- |
| 保留旧版发票支持，扩展数电票 | ✅ 完全采纳 | 现有 10 个 schema 全部保留，旧票种一条不动 |
| 用官方标准对齐字段（什么字段是什么） | ✅ 完全采纳 | `xbrlJsonConfig` 20 个配置是官方字段字典，尤其金额含税语义（数电三项不含税、铁路 Fare 含税、航空 Fare 不含税） |
| 以官方 JSON 样例作为**提取模板** | ⚠️ 需修正 | 实测：官方**接收端** JSON（einv_ord/spcl_receiver）**没有明细行字段**——它面向报销入账流程，只记总额+会计状态。而我们的台账恰恰按明细行组织。照搬会丢掉整张明细表 |
| 用官方样例做验证 | ✅ 完全采纳 | 样例每个票种仅 1 个实例、且部分为 2021-2022 年旧场景数据——它的正确角色是 **golden test 基准**，不是模板 |

**修正后的表述**：以官方**字段配置**为语义标准，以**税务总局交付 XML 的结构**（含 `IssuItemInformation` 明细行，高德样本实测齐全）为数电票目标模型，以官方 JSON 样例为验证基准。三者各司其职，不是单靠样例。

另一个必须说清的事实：JSON 样例里没有版面信息（别名、位置、正则），**PDF 识别规则无法从样例"模板化"生成**——PDF 路径的 schema 仍需按真实票面维护，只是字段语义按官方字典校准。

## 二、总体架构：双轨制

```
输入文件
  │
  ├─ A. 结构化数据通道（新增，零识别误差）
  │     ├─ A1. PDF 内嵌 XBRL？→ 官方 config 驱动解析（铁路已实测 2/2）
  │     └─ A2. 独立 XML 文件（总局 EInvoice 格式，如高德交付）
  │           → 同一解析器，复用同一映射（含明细行）
  │
  └─ B. PDF 文本层 / OCR 识别通道（保留 + 校准）
        ├─ 有官方标准的票种（数电、铁路、航空、通行费、财政票据）：
        │   字段语义按官方字典校准（P1a 航空修复为首个落地）
        ├─ 无官方标准的旧票种（出租车、定额、通用机打等）：
        │   现有 schema 原样保留
        └─ 回归验证：官方样例 + 真票内嵌数据逐字段对拍
```

两轨产出统一写入现有 InvoiceRecord → 台账模板不变、写入与去重逻辑不动。

## 三、票种覆盖矩阵（每类票走哪条路）

| 票种 | 主路径 | 目标模型来源 | 备注 |
| --- | --- | --- | --- |
| 数电普票/专票 | A2（有 XML 时）→ B（只有 PDF 时） | 总局 XML 结构 + einv 语义 | PDF 实测不内嵌，PDF 路径必修 |
| 铁路电子客票 | **A1 内嵌 XBRL 直读** | rai 官方字段（售/退、税率、税额齐全） | 无内嵌时 B 兜底 + 公式标注 |
| 航空行程单 | B（PDF 实测不内嵌） | atr 官方字段校准（Fare/Fuel/VatTaxAmount/民航基金） | P1a 修复即按此模型 |
| 通行费发票 | B | inv_tlf 官方字段（工具包已备） | catalog 缺口，借工具包补齐 |
| 财政票据/医疗票据 | B（efi 语义参考） | efi 官方字段 | 无内嵌样本，留 A1 接口不宣称 |
| 出租车/定额/机打等旧票 | B（现状保留） | 无国家标准，现有 schema | 不动 |
| 银行回单/对账单、国库凭证 | 不做 | — | 超出发票台账边界 |

## 四、落地步骤（按依赖排序）

1. **配置入库**：工具包 20 个 `xbrlJsonConfig/*.xml` 复制为 `config/xbrl/`（注明来源：财政部基础工具包 V1.0，taxonomy 2023-12-31）
2. **解析器**：新增 `input_profile/embedded_xbrl.py`——`embfile` 检测 → 命名空间/configId 匹配（按元素名解析、与 taxonomy 版本解耦）→ 映射 InvoiceRecord。约 100 行，零新依赖
3. **铁路票切换 A1**：乘车票读官方税额；退票费按官方 `TypeOfBusiness=退` + 6% 值写入，备注提示抵扣确认（P1b 落地）
4. **航空修复 P1a**：schema 拆 digital/legacy 变体，数电版按 atr 官方语义做两行明细（票价+燃油/9%/票面税额；民航基金不计税），简单相加对齐合计
5. **独立 XML 输入 A2**：`--input` 接受 `.xml`，总局 EInvoice 格式走同一解析器（含 `IssuItemInformation` 明细行映射）
6. **P2 修复**：`is_positive_invoice` 按金额符号派生；A 轨有官方红蓝标志字段时优先用官方值
7. **golden test**：官方 17 个 JSON 样例 + 三张真票内嵌数据作为回归基准，B 轨输出逐字段对拍
8. **其余 REVIEW.md 顺手修项**（P6~P11）照常

## 五、明确不做（防过度设计）

- 不引入 Java 运行时与工具包 jar（配置是数据，30 行 Python 实证可复刻）
- 不做 XBRL→官方 JSON 中间层（直映射 InvoiceRecord）
- 不取接收端会计入账字段（分录、三单匹配、抵扣属期）
- 不做 SM2 验签、云查验
- 不做 OFD 解析（无样本不宣称；OFD 本身是 zip，留接口）
- 不做银行/国库类凭证（超出发票台账边界）

## 六、一句话总结

**旧票靠 B 轨原样保留；数电票能直读就直读（A 轨官方数据），不能直读就按官方语义校准着识别（B 轨），官方样例负责验证对错。** 全部零新依赖，改动集中在 6 个文件 + 1 个新模块。

## 附录：实施前准备（护栏，半天内可完成）

动工不阻塞，但以下 4 项先定好，保证实施严格按本方案走、可验收：

### 1. 建立回归基线（动工前第一件事）
用当前未修改的 skill 对 `数电发票/` 全部真票（36 张）跑一遍，存档输出（台账副本 + run_summary）作为"改造前基线"。之后每一步修改都重跑同一批票对比：哪些行变了、变对还是变错，一目了然。**没有基线就没有"严格按方案实施"的判据。**

### 2. 双轨一致性约束（写进验收标准）
同一张票走 A 轨（官方数据）或 B 轨（版面识别）必须产出**相同的 invoice_line_key**（号码、日期、价税合计、销方税号语义一致），否则同一发票重复入账。每条轨道的输出字段语义以官方字典为准——这不是新需求，是把"双轨制"说完整的必要约束。

### 3. 配置入库范围拍板（建议比方案更克制）
方案写"20 个配置全入库"，建议改为**只入有落地路径的 8 个**（rai_issuer/receiver、atr_issuer/receiver、einv_ord/spcl_receiver、inv_tlf_receiver、efi），其余 12 个挂清单备注来源，有真实样本时再入。理由：没有样本验证的配置入库即死资产，违背"没有样本不宣称"原则。

### 4. 两个需要用户拍板的决策点
- **台账模板加列**（P7 明细 sheet 补"开票日期"列）：改模板 xlsx 需要用户确认——加了列旧台账要同步加列才能继续追加
- **实施载体**：在源仓库（E:\maple192600…）改并验证后，同步到已安装副本（…\.workbuddy\skills\invoice-ledger）。建议单一事实源在源仓库，安装目录只做同步，避免两处漂移

### 动工顺序（不变，1-2 步当天可完成）
配置入库（按第 3 项的 8 个）→ `embedded_xbrl.py` → 铁路票切 A1 并过基线对比 → 航空 P1a → A2 独立 XML → P2 → golden test → 顺手修项。每步验收 = 回归基线对比 + 官方样例对拍。
